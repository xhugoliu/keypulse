# 产品简报

## 问题

键盘固件能比主机应用更准确地观察物理按键活动。主机输入 API 看到的是已经被操作系统处理过的结果：按键事件、文本、快捷键、输入法效果或系统级抽象。KeyPulse 关注的是更底层的物理事实：哪个开关位置发生了变化，什么时候变化，当时处在哪一层，以及按键被按住了多久。

早期实验项目 `minix-insight` 已经证明了这条路径在一把 QMK 键盘上是有价值的。KeyPulse 的目标是把这个能力推广到多种固件体系和跨平台桌面应用。

## 目标用户

- 希望查看使用热力图和人体工学反馈的键盘用户。
- 希望复用按键数据适配器的固件玩家。
- 希望用物理使用数据调优布局的键盘设计者。
- 想比较层、组合键、拇指键、左右手负载和 hold-tap 行为的重度用户。

## 目标

- 从 QMK 和 ZMK 固件采集物理按键活动。
- 把不同 transport 上来的事件归一化为同一个主机侧模型。
- 通过 Tauri 支持 macOS 和 Windows 桌面接收端。
- 先接入 miniX 和 pskeeb5，再扩展到 X.Tips QMK/Vial 固件集合里的其他型号。
- 默认本地优先、透明可控。
- 在不过度设计第一版应用的前提下，为未来固件适配器保留足够清晰的协议结构。

## 非目标

- 记录用户输入文本。
- 重建密码、词句或应用内容。
- 依赖 macOS Accessibility API 或 Windows 键盘 hook。
- 做云端分析产品。
- 替代 VIA、Vial、ZMK Studio 或固件配置器。
- 第一版支持所有平台。
- 兼容旧的 `KS` Raw HID 报文。

## MVP

MVP 达成标准：

- 启用 KeyPulse 的 miniX QMK 固件能通过 Raw HID 发送 KeyPulse 事件。
- Tauri 桌面应用能在 macOS 和 Windows 上接收事件。
- 事件能带着主机时间戳写入本地存储。
- miniX profile 能渲染物理热力图和基础日统计。
- 协议和 profile 模型没有写死在 miniX 上。
- pskeeb5 的 ZMK 适配路径已经清楚到可以开始实现。

## 产品原则

- 物理优先：先保存位置和矩阵事实，再保存派生标签。
- 本地优先：默认应用不会把按键数据发送到本机之外。
- 适配器友好：增加一把键盘不应该要求重写应用。
- Transport 无关：HID、serial 或未来 BLE 应承载同一套事件语义。
- 状态诚实：接口占用、协议不支持、profile 缺失都应该在 UI 中清楚可见。
